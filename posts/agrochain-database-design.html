My AgroChain Project - Database Design Explained

March 15, 2026 8 min read

When I started working on AgroChain, my Java-based agricultural supply chain management system, I knew database design would make or break the project. A poorly designed database leads to data redundancy, integrity issues, and a nightmare to maintain. In this post, I'll walk you through how I approached the database design for AgroChain.

The Problem Domain

AgroChain tracks agricultural products from farm to consumer. The key entities I needed to model were:

  • Farmers - Who grows the produce
  • Products - What is being grown and harvested
  • Harvests - Individual harvest events
  • Distributors - Who transports products
  • Retailers - Who sells to consumers
  • Transactions - Every transfer of ownership

Entity Relationships

The first step was identifying how these entities relate to each other:

Farmer to Harvest: One farmer can have many harvests (1:N). A single farmer grows multiple crops throughout the year.

Harvest to Product: Many harvests produce the same product type (N:1). Multiple batches of tomatoes from different farms are still tomatoes.

Harvest to Transaction: One harvest can have multiple transactions as it moves through the supply chain (1:N). Each sale from farmer to distributor, distributor to retailer, and retailer to consumer is a separate transaction.

Transaction to Distributor/Retailer: Each transaction involves one buyer and one seller from these entities (N:1).

Normalization: Reaching Third Normal Form

I designed the database to at least Third Normal Form (3NF) to eliminate redundancy and ensure data integrity:

First Normal Form (1NF): All columns contain atomic values. No repeating groups. Each product name is a single value, not a comma-separated list.

Second Normal Form (2NF): No partial dependencies. All non-key attributes depend on the entire primary key. In the harvest table, farmer_id and harvest_date together form the key, and all other attributes depend on both.

Third Normal Form (3NF): No transitive dependencies. A farmer's name doesn't depend on their ID through the harvest. I created separate tables to avoid this.

The Database Schema

Here's a simplified view of my final schema:

-- Core Tables
farmers (
    farmer_id (PK),
    name,
    email,
    phone,
    address,
    created_at
)

products (
    product_id (PK),
    name,
    category,
    unit,
    created_at
)

harvests (
    harvest_id (PK),
    farmer_id (FK),
    product_id (FK),
    quantity,
    harvest_date,
    quality_grade,
    created_at
)

distributors (
    distributor_id (PK),
    name,
    license_number,
    address,
    contact_person,
    created_at
)

retailers (
    retailer_id (PK),
    name,
    location,
    contact,
    created_at
)

transactions (
    transaction_id (PK),
    harvest_id (FK),
    transaction_type (FARMER_TO_DISTRIBUTOR,
                     DISTRIBUTOR_TO_RETAILER,
                     RETAILER_TO_CONSUMER),
    seller_type (distributor/retailer),
    seller_id,
    buyer_type,
    buyer_id,
    quantity,
    price,
    transaction_date,
    blockchain_hash,
    created_at
)

Blockchain Integration

What makes AgroChain special is the blockchain integration. Each transaction gets a cryptographic hash that links to the previous transaction, creating an immutable audit trail. The blockchain_hash field stores this hash.

When a new transaction is created, my Java application:

  1. Fetches the previous transaction's hash
  2. Creates a hash of current transaction data
  3. Links them together
  4. Stores the new hash in the blockchain_hash field

Indexes for Performance

I added indexes on frequently queried columns:

  • harvests(farmer_id, product_id) - For filtering by farmer or product
  • transactions(harvest_id) - For tracing a product's journey
  • transactions(transaction_date) - For reports by time period
  • products(category) - For category-based filtering

Challenges Faced

One challenge was handling the polymorphic nature of transactions. A transaction can involve either a distributor or retailer as the seller. I solved this with the seller_type and seller_id fields, which together reference the appropriate table.

Another challenge was ensuring data consistency across the blockchain. If a transaction is updated, it breaks the chain. I implemented triggers to prevent updates to committed transactions, making the blockchain truly immutable.

Lessons Learned

Database design is 80% planning and 20% implementation. Spending time on paper (or in a tool like dbdiagram.io) before writing SQL saved me from major refactoring later. The entity-relationship diagram became my roadmap.

Normalization isn't always the answer for performance. In production, I might denormalize certain read-heavy queries. But for a learning project like this, starting normalized and understanding why you'd denormalize is the right approach.

If you're working on a similar project, I recommend starting with the entities, drawing relationships, normalizing to 3NF, then optimizing based on your actual query patterns. The upfront investment pays off in maintainability.
