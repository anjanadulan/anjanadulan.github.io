How I Take Notes for Programming Courses

February 10, 2026 6 min read By Anjana Wijerathna

Note-taking for programming is different from other subjects. Here's my system for capturing code concepts, syntax, and explanations that actually helps during exams and projects.

Why Programming Notes Are Different

In literature or history, notes are mostly text - concepts, dates, arguments. In programming, you need to capture code, which has its own syntax and structure. A page of text describing loops isn't as useful as seeing actual loop examples.

My note-taking system evolved to handle both: conceptual explanations and practical code snippets. It's not perfect, but it's working for me.

The Tools I Use

Notion: My primary note-taking app. It handles text, code blocks, databases, and has excellent organization features. The ability to embed code with syntax highlighting is essential.

VS Code: For code-specific notes and snippets. I'll create small test files to understand concepts, then keep useful snippets in organized folders.

Physical Notebook: Surprisingly useful for diagrams, flowcharts, and quick sketches during lectures. Something about writing by hand helps me remember algorithms.

My Note Structure

Each topic follows a consistent structure:

  1. Concept Definition: What is it? A clear, plain-English explanation.
  2. Why It Matters: When would you use this? What problem does it solve?
  3. Code Example: A working, minimal example that demonstrates the concept.
  4. Syntax Reference: The exact syntax with explanations of each part.
  5. Common Mistakes: Pitfalls I've encountered or heard about.
  6. Practice Problems: Exercises to reinforce learning.

Taking Notes During Lectures

Lectures are for listening, not transcribing. My approach:

Before class: Skim the textbook chapter or slides. Not in depth - just enough to know what topics are coming. This primes my brain to recognize the concepts.

During class: Focus on understanding. Write down key concepts, not everything the professor says. If they write code on the board, I photograph it rather than copy it out - I can annotate the photo later.

After class: Process the notes within 24 hours. This is crucial. Rewrite them, add code examples from my own experimentation, and connect to previous topics.

Code Examples: The Most Important Part

For every concept, I include code. Not pseudo-code, not descriptions - actual working code.

My code examples follow this format:

// Concept: Variable Scope
// Topic: JavaScript Basics

// Global scope - accessible everywhere
const globalVar = "I'm global";

function demonstrateScope() {
    // Function scope - only accessible inside function
    const functionVar = "I'm in function";
    console.log(globalVar); // Works
    console.log(functionVar); // Works
    
    if (true) {
        // Block scope - only inside this block
        let blockVar = "I'm in block";
        const alsoBlock = "Me too";
        console.log(blockVar); // Works
    }
    
    console.log(blockVar); // Error! Block variables not accessible
}

The comments explain what's happening. I can read the code and understand it without needing external context.

Organizing Your Notes

I use Notion databases to organize notes by:

  • Course: Which class is this for?
  • Topic: Algorithm, Database, Web Development, etc.
  • Status: In Progress, Reviewed, Mastered
  • Priority: High (exam soon), Medium, Low

This makes it easy to find notes when studying and track what I've covered vs. what needs review.

The Review System

Notes without review are almost useless. I use spaced repetition:

Day 1: Take initial notes during/after class.

Day 3: First review - can I explain the concept without looking?

Week 1: Second review - code the example from scratch.

Month 1: Final review before exams.

If I can't explain something at each stage, I mark it for extra attention.

What Doesn't Work (For Me)

Writing everything verbatim: I tried this. It's tedious, I don't retain it, and I miss understanding the concepts while trying to keep up with typing.

Only watching video tutorials: Videos are passive. Even with note-taking, I learn better by doing. I'll watch a video, then immediately try to code the same thing without looking.

Messy organization: When notes are disorganized, I don't review them. Structure encourages consistency.

Tips for Better Programming Notes

Include error messages: When learning a concept, include common errors and how to fix them. "If you get this error, it usually means..."

Compare similar concepts: Arrays vs ArrayLists, let vs const, for vs while. Side-by-side comparisons clarify differences.

Write your own explanations: Don't just copy textbook definitions. Explain concepts in your own words. If you can't explain it simply, you don't understand it.

Test the code: Every code example should be tested. Don't include code you're not sure works. Nothing worse than studying incorrect code.

Final Thoughts

Note-taking is personal. What works for me might not work for you. The key is experimenting and being honest about what you actually use vs. what looks good in theory.

Start simple: try one tool, one structure, and give it a few weeks. If you're not using your notes, something needs to change.
