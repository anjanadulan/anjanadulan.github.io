Getting Started with Git and GitHub: A Beginner's Guide

January 10, 2026 8 min read By Anjana Wijerathna

Version control is a fundamental skill for every developer. Here's my practical guide to understanding Git and GitHub, even if you've never used them before.

What Even Is Version Control?

Let me paint a familiar picture. You've been working on a project for hours. You make a change, realize it broke something, and spend the next hour trying to remember what the code looked like before. You create files named "final_v2_FINAL" and "final_v3_REALLY_FINAL_THIS_TIME" and eventually lose track of which is which.

Version control solves this. It's like an unlimited undo button for your entire project, combined with the ability to create parallel versions of your code that you can later merge together.

Git is the most popular version control system. It's a command-line tool that tracks changes to your files locally. GitHub is a cloud platform that hosts your Git repositories online, making collaboration and backup easy.

Setting Up Git

First things first: install Git. On Windows, download it from git-scm.com. On Mac, you can use Homebrew with brew install git. On Linux, use your package manager.

After installation, configure your identity:

git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

This information gets attached to every commit you make. It's how others know who wrote what code.

Basic Concepts You Need to Understand

Repository (repo): A folder that Git is tracking. Contains your project files and a special .git folder that stores all the version history.

Commit: A snapshot of your changes at a specific point in time. Think of it as saving your game progress - you can always return to that save point.

Branch: A separate line of development. The main branch is usually called "main" or "master". Creating a branch lets you experiment without affecting the main code.

Merge: Combining changes from one branch into another. This is where the magic happens - bringing together different features.

Your First Repository

Let's create a repository from scratch:

# Create a new folder and enter it
mkdir my-project
cd my-project

# Initialize a new Git repository
git init

# Create a file
echo "Hello, Git!" > hello.txt

# Check the status
git status

Running git status shows you that Git sees your new file but isn't tracking it yet. That's because we haven't added it to the staging area.

The Staging Area: Understanding git add

Git has three main areas:

  • Working Directory: Where your actual files live
  • Staging Area: A holding area for changes you want to commit
  • Repository: Where committed changes are stored permanently

The workflow is: modify files in the working directory, add them to staging, then commit from staging to the repository.

# Stage a specific file
git add hello.txt

# Or stage everything
git add .

# Commit with a message
git commit -m "Add hello.txt with greeting"

Branching: Your Safety Net

Here's where things get powerful. Say you want to add a new feature but aren't sure if it'll work. Create a branch:

# Create and switch to a new branch
git checkout -b feature/new-feature

# Do your work, commit changes
git add .
git commit -m "Start working on new feature"

# Switch back to main
git checkout main

# See all branches
git branch

The main branch stays clean while you experiment. When you're happy with your feature, merge it back:

# Switch to main
git checkout main

# Merge your feature branch
git merge feature/new-feature

# Delete the feature branch when done
git branch -d feature/new-feature

GitHub: Taking It Online

Create a GitHub account at github.com. Then, create a new repository there. You'll get instructions to push your local repo to GitHub:

# Add the remote repository
git remote add origin https://github.com/username/repo.git

# Push your code
git push -u origin main

The -u flag sets up tracking so future pushes just need git push.

The Collaboration Workflow

When working with others on GitHub:

  1. Fork the repository (create your own copy)
  2. Clone it to your computer
  3. Create a branch for your changes
  4. Make and commit your changes
  5. Push to your fork
  6. Create a Pull Request (PR) on GitHub
  7. Wait for review and merge

This workflow is how open-source projects accept contributions. I've contributed to several projects this way, and it's incredibly satisfying to see your code merged into a project used by thousands of people.

Common Commands I Use Daily

After months of using Git, here's my essential command vocabulary:

git status          # What changed?
git log --oneline  # Commit history
git diff           # What exactly changed?
git stash          # Temporarily save changes
git pull           # Get latest from remote
git fetch          # Download but don't merge

Mistakes I Made (So You Don't Have To)

Committing sensitive data: Never commit API keys, passwords, or tokens. Use environment variables and .gitignore files.

Committing large files: Git struggles with files over 50MB. Use .gitignore to exclude build outputs, node_modules, and binaries.

Not writing good commit messages: "Fixed stuff" tells no one anything. "Add user authentication with JWT tokens" is useful.

Pushing to main directly: Always use branches for features. It's not about being cautious; it's about developing good habits.

Final Thoughts

Git and GitHub might seem overwhelming at first, but they're essential tools that become second nature with practice. Start with the basics: init, add, commit, push, pull. Branching comes naturally after that.

My best advice? Use Git for every project, even personal ones. The small projects you don't think matter are where you'll learn the most, because there are no stakes and no pressure.
